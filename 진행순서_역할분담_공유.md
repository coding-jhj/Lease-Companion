## 2. 전체 진행 순서 (한눈에)

```
[0단계] 터 닦기
   모두의 공통 재료를 먼저 만든다
        ↓
[1단계] 각자 기초 공사
   A: 통합 스키마 코드화   B: 회원·계약 건 API   C: 화면 뼈대
        ↓
[2단계] 각자 본 공사
   A: RAG(근거 검색)      B: 저장·이력·연결      C: 신호등 UI·대시보드
        ↓
[3단계] 연결 공사
   A: 질문·행동 생성      B: 남은 API 마무리     C: 진짜 서버와 연결
        ↓
[4단계] 다 같이 통합 검증
   CASE-001로 처음부터 끝까지 돌려보기
        ↓
[5단계] J01~J12 전체 판정 확장
```

### 현재 진행 상태 (2026-07-16)

| 담당 | 현재 상태 | 다음 확인 |
|---|---|---|
| A | 1단계 완료 — canonical v1.1.0·RuleResult 13필드·실제 AI 경로·JSON Schema 5개·CASE-001 fixture 6개·전체 테스트 148개 | 공식자료 검증 결과를 기반으로 RAG 구현 시작; B·C 소비 체크는 아직 미완료 |
| B | PostgreSQL 설정과 회원·계약 건·계약 상황 API 구현 | A 모델 import·fixture 검증·저장 왕복 등 인수 체크리스트 수행 |
| C | 프로젝트 초기화 전 | React+Vite+TypeScript 초기화 후 fixture 소비 체크리스트 수행 |

최종 인수인계 완료는 `docs/api/data-contract-v1.md`의 B·C 체크리스트를 각 담당자가 통과한 시점이다.

## 3. 역할 분담

### 담당 A — AI·데이터 담당

"문서에서 뽑은 데이터를 제대로 된 모양으로 만들고, 근거와 설명을 붙이는 사람"

| 순서 | 할 일 | 결과물 |
|---|---|---|
| 1단계 | **통합 스키마를 코드로 만들기** — 합의된 표(ExtractedField, 판정 코어)를 `ai/src/lease_companion_ai/schemas/`에 Pydantic 모델로 작성. 기존 코드가 새 모양을 쓰도록 어댑터 연결. 바꾸기 전·후 규칙 결과가 같은지 회귀 테스트 | 모두가 쓰는 데이터 모양 확정 |
| 2단계 | **RAG 만들기** — 법령·공식 자료를 잘라서(청킹) Chroma에 넣고, 규칙 결과마다 관련 근거를 검색해 연결. 임베딩은 gemini-embedding-001, 재정렬은 Cohere | 판정마다 "이 법령 근거" 자동 연결 |
| 3단계 | **질문·행동 생성 + Guardrail** — GPT로 확인 질문·체크리스트·행동 문구 생성. 단정 표현("안전합니다" 등)과 근거 없는 출력은 차단 | 규칙 결과 기반 안내문 |

주의: 생성(3단계)은 반드시 RAG(2단계) 다음. 근거 없이 생성부터 만들면 원칙 위반.

### 담당 B — Backend·저장 담당

"회원과 계약 건 데이터를 받아서 PostgreSQL에 저장하고 다시 꺼내주는 사람"

| 순서 | 할 일 | 결과물 |
|---|---|---|
| 0단계 | **PostgreSQL 설치·연결 셋업** (Docker 또는 로컬 설치) + SQLAlchemy 연결 확인 | 팀 공용 DB 준비 |
| 1단계 | **회원 API** (signup / login / me — JWT 발급) + **계약 건 API** (생성·목록·상세) + **계약 상황 입력 API** | 로그인하고 계약 건을 만들 수 있음 |
| 2단계 | **문서 업로드 저장** (계약 건에 연결, 이력 관리) + **모의 등기 연결** (`registry-link`, case_id 기준) + **분석 결과 저장·재조회** (버전·재분석 이력) | 분석 결과가 사라지지 않고 다시 열림 |
| 3단계 | **체크리스트·계약 직후 행동 API** (상태 저장·재조회) + 접근 통제 (남의 계약 건 못 보게) | 8단계 흐름의 저장 기능 완성 |

주의: 추출 결과 **저장 모델**은 A의 스키마(1단계) 확정 후 작성. 그 전에 만들면 두 번 일하게 됨.

### 담당 C — Frontend 담당

"사용자가 보는 화면을 React로 만드는 사람"

| 순서 | 할 일 | 결과물 |
|---|---|---|
| 0단계 | **React + Vite + TypeScript 프로젝트 셋업** (`frontend/`에) + 라우팅 뼈대 | 개발 시작 준비 |
| 1단계 | **8단계 화면 뼈대** — 로그인/가입 → 대시보드 → 계약 건 생성 → 상황 입력 → 업로드 → 확인·수정 → 리포트 → 체크리스트. **가짜 데이터(mock)로 개발** — API 목록의 응답 모양대로 임시 데이터를 만들어 쓰면 B를 기다릴 필요 없음 | 화면 흐름 전체가 클릭으로 이어짐 |
| 2단계 | **신호등 UI** — 반드시 확인(빨강 계열)·확인 권장·일반 확인 3그룹. 색만 쓰지 말고 문구·아이콘도 함께 (색약 사용자 배려) + 로딩·오류·빈 화면 상태 + 대시보드에서 기존 리포트 다시 열기 | 결과 화면 완성 |
| 3단계 | **mock을 진짜 API로 교체** — B의 서버에 하나씩 연결. 모양을 처음부터 맞춰놨으므로 교체만 하면 됨 | 실제로 동작하는 앱 |

주의: "추출값 확인·수정 화면"의 데이터 모양은 A의 스키마를 따름. 1단계 확정 전에는 다른 화면부터 진행.

## 4. 서로 지켜야 할 규칙 3가지

1. **A의 스키마가 끝나기 전에, 추출 데이터를 저장(B)하거나 그리는(C) 코드를 만들지 않는다.**
   → 만들면 스키마 확정 후 전부 다시 고쳐야 함. 그동안 B는 회원·계약 건, C는 로그인·대시보드처럼 스키마와 무관한 것을 진행.
2. **API의 요청·응답 모양을 바꾸고 싶으면 반드시 셋이 공유 후 변경.**
   → 한 명이 조용히 바꾸면 mock으로 개발하던 C의 화면이 연결할 때 깨짐.
3. **새 API가 필요해지면 목록에 추가하는 건 자유, 기존 것을 바꾸는 건 신중하게.**

## 5. 4단계 — 통합 검증 (셋이 함께)

CASE-001로 처음부터 끝까지 실제 시나리오를 돌려본다:

- [ ] 회원가입 → 로그인 → 계약 건 생성 → 상황 입력 → 문서 업로드 → 추출 → 수정·확인 → 분석 → 리포트 → 체크리스트
- [ ] 로그아웃 후 다시 로그인해도 리포트가 다시 열리는가
- [ ] 다른 계정으로 로그인하면 남의 계약 건이 안 보이는가
- [ ] Gemini 키를 빼고 돌려도 (폴백 경로) 앱이 죽지 않는가
- [ ] 로그에 이름·주소 같은 개인정보가 찍히지 않는가
- [ ] "안전합니다" 같은 단정 문구가 화면 어디에도 없는가

통합 검증을 통과한 뒤 5단계에서 J01~J12 전체 판정으로 확장한다. R01~R10과 J01~J12는 다른 축이며 기존 R 필드 이름·의미는 유지한다.

## 6. 용어 풀이 (헷갈릴 때)

| 용어 | 쉬운 뜻 |
|---|---|
| 스키마 | 데이터의 "모양 약속". 어떤 이름의 칸에 어떤 종류의 값이 들어가는지 정한 것 |
| Pydantic | 파이썬에서 그 약속을 코드로 쓰고, 어긋난 데이터가 오면 잡아주는 도구 |
| mock | 진짜 서버 대신 쓰는 가짜 응답. 화면 개발을 서버보다 먼저 시작하게 해줌 |
| JWT | 로그인 성공 시 서버가 주는 출입증. 이후 요청마다 이걸 보여줌 |
| RAG | 질문과 관련된 문서를 찾아서(검색) 근거로 붙이는 기술 |
| Guardrail | AI가 하면 안 되는 말(단정·근거 없는 말)을 걸러내는 안전장치 |
| 회귀 테스트 | 코드를 고친 뒤에도 예전과 같은 결과가 나오는지 확인하는 테스트 |
| ADR | "우리 이 기술 쓰기로 했고 이유는 이것" 을 남기는 결정 기록 문서 |
