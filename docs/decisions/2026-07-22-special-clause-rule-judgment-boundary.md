# 특약 카탈로그와 R/J 판정 책임 경계

- 상태: 확정
- 날짜: 2026-07-22
- 대상: AI·Backend·Frontend 구현자
- 적용 계획: 혼자 작업용 특약 RAG 작업 3 이후

## 배경

특약 카탈로그는 확인된 특약 원문에서 여섯 유형의 후보와 공식 근거 검색 범위를 고른다.
기존 R01~R24와 J01~J12는 이미 저장·API·화면에서 사용하는 판정 축이다. 카탈로그가 불리한
문구를 찾았다는 이유만으로 R/J 상태나 시급도를 만들면 기존 판정 의미가 바뀌고, LLM·RAG가
Python 최종 판정을 우회하게 된다.

특히 다음 경계가 불명확했다.

- R08의 반환 조건 명확성과 J10의 미래 사건 연동 확인 신호
- R09·J11 안에서 계약 중 수리와 계약 종료 시 원상복구를 구분하는 방법
- R10의 권리변동 제한 문구와 R19의 실제 시점별 권리변동 확인
- J12와 별도의 신규 R 규칙을 만들지 여부

## 결정

기존 R01~R24와 J01~J12 식별자·사용자 표시명은 유지한다. 특약 카탈로그는 후보 문맥만
제공하고 `RuleStatus`·`urgency`·최종 `reason`을 만들거나 변경하지 않는다.

### R 규칙 경계

| 규칙 | 유지하는 의미 | 입력·상태 경계 | 특약 카탈로그 역할 |
|------|---------------|----------------|--------------------|
| R08 | 보증금 반환 시점·조건 명확성 | 기존 명확성 입력을 사용해 `명확`·`불명확`·`미기재`·`확인 필요`만 반환 | `SC-DEFERRED-REFUND` 후보가 관련 문맥을 제공하지만 불리 조건의 최종 상태는 만들지 않음 |
| R09 | 수리·원상복구 책임 명확성 | 책임 주체·범위·비용의 기재 명확성만 확인 | `SC-REPAIR-SCOPE`와 `SC-RESTORATION-SCOPE`가 논점을 구분하되 R09 상태를 직접 결정하지 않음 |
| R10 | 계약 후 권리변동 제한 특약 유무 | 계약서에 담보권 설정 제한 문구가 있는지만 `명확`·`미기재`·`확인 불가`로 확인 | `SC-RIGHTS-CHANGE`가 제한 문구 후보를 제공할 수 있음 |
| R19 | 계약 전후 실제 권리변동 확인 | 계약·잔금·입주 시점별 등기사항증명서 자료로 변동 여부를 확인. 필요한 시점 자료가 없으면 자동 완료하지 않고 기존 명세의 `확인 불가` 경계를 사용 | 카탈로그 문구만으로 R19를 `명확` 또는 `미기재`로 만들지 않음 |

작업 4에서 R19가 `rights_change_clause_present`를 사용하던 중복 구현을 제거했다. 현재
canonical 입력에는 시점별 등기 이력이 없으므로 신규 분석의 R19는 `확인 불가`·`분석 불가`를
반환한다. 과거 저장 결과를 읽기 위한 기존 허용 상태는 유지하며, 향후 시점별 자료 입력은
additive schema 변경으로 연결한다.

### J 판정 경계

| 판정 | Python 최종 판정 책임 | 허용 상태 | 특약 후보 사용 방법 |
|------|-----------------------|-----------|---------------------|
| J10 | 반환 시점·조건의 명확성과 명시적 미래 사건 연동을 확인 | `명확`·`불명확`·`미기재`·`확인 필요` | 확인 원문과 classification 후보를 함께 사용. 신규 임차인 입주·주택 매각·임대인 자금 사정에 반환을 연동하면 문장이 명확해도 `확인 필요`; `관계없이` 등 명시적 독립 문구는 제외 |
| J11 | 수리·원상복구 책임 주체와 범위의 명확성을 하나의 최종 상태로 확인 | `명확`·`불명확`·`미기재`·`확인 필요` | 카탈로그 유형으로 계약 중 수리와 종료 시 원상복구 논점을 구분하고 `reason`에서 해당 논점을 설명 |
| J12 | 본문과 특약의 동일 주제 조건을 결정적으로 비교 | `상충 가능`·`명확`·`확인 필요`·`적용 제외` | `SC-MAIN-SPECIAL-CONFLICT`가 후보 주제를 제공. Python이 날짜·금액·반환 조건·책임 주체를 비교해 최종 상태 결정 |

J10의 신규 임차인 조건과 독립 문구는 현재 구현되어 있다. 주택 매각·임대인 자금 사정 조건,
J11 논점별 `reason`, J12 비교 범위 보강은 작업 4에서 구현한다.

### 신규 필드·규칙

- R08·R09의 기존 의미를 다른 위험 판정으로 바꾸지 않는다.
- `repair_responsibility`·`restoration_responsibility` 같은 신규 canonical 하위 필드는 작업 3에서 추가하지 않는다.
- J12 대응용 R25를 추가하지 않는다. 본문·특약 상충의 최종 판정은 J12가 담당한다.
- 신규 하위 필드는 현재 입력으로 결정론적 판정이 불가능하다는 테스트 근거가 생길 때 별도 schema ADR로 검토한다.

## 카탈로그 연결 계약

| 카탈로그 유형 | 관련 R | 관련 J |
|---------------|--------|--------|
| `SC-DEFERRED-REFUND` | R08 | J10 |
| `SC-REPAIR-SCOPE` | R09 | J11 |
| `SC-RESTORATION-SCOPE` | R09 | J11 |
| `SC-RIGHTS-CHANGE` | R10·R19 | 없음 |
| `SC-MANAGEMENT-FEE` | R18 | J09 |
| `SC-MAIN-SPECIAL-CONFLICT` | 없음 | J12 |

이 연결은 관련 판정 결과를 카드에 연결하기 위한 참조다. 카탈로그 매칭 성공 자체는 상태나
시급도의 근거가 아니다. 미매칭 문장은 새 판정을 만들지 않는다.

## 처리 순서

1. 사용자 확인 완료 `InputSnapshot`에서 특약 원문을 읽는다.
2. Classification은 조항 유형·명확성·책임 주체·조건 후보만 만든다.
3. 특약 카탈로그는 유형 후보와 허용 source·section 범위를 고른다.
4. Python R/J 규칙이 최종 상태·시급도·판정 이유를 결정한다.
5. 특약 RAG는 확정 상태를 바꾸지 않고 공식 근거만 검색한다.
6. 생성·Guardrail은 근거 범위 안에서 설명·확인 질문·수정 요청 문구를 만든다.

## 하위 호환

- 기존 R/J 식별자와 사용자 표시명을 유지한다.
- 기존 저장 결과는 생성 당시 결과로 보존하며 새 의미로 재해석하거나 자동 재계산하지 않는다.
- 신규 실행에서 J10/J11이 있으면 사용자 안내는 각각 R08/R09보다 J 결과를 우선해 중복 안내를 만들지 않는다.
- R19 입력을 시점별 등기 자료로 보완할 때 필요한 필드는 additive schema 변경으로만 검토한다.
- R19 전환 시 rule spec·fixture·OpenAPI·저장 결과 호환을 함께 검증하고 schema version 변경 필요성을 별도 판단한다.

## 검토한 대안

### R08·R09의 의미를 불리 조건 판정으로 변경

기존 저장 결과와 안내 의미가 달라지고 J10·J11과 중복되므로 채택하지 않았다.

### 수리와 원상복구마다 신규 J 또는 canonical 필드 추가

현재는 카탈로그 유형과 `reason`으로 논점을 구분할 수 있다. 새 입력 계약의 필요성이 테스트로
확인되지 않았으므로 채택하지 않았다.

### 본문·특약 상충용 R25 추가

J12가 이미 같은 최종 판정 책임을 갖고 있어 중복 축이 되므로 채택하지 않았다.

### R10과 R19를 같은 특약 유무 규칙으로 유지

문구 존재와 실제 등기 변동은 다른 사실이다. 같은 입력으로 두 결과를 완료하면 사용자가
실제 등기 변동까지 확인된 것으로 오해할 수 있어 채택하지 않았다.

## 영향과 후속 작업

- 작업 4: 미래 사건 표현, 부정·예외·복합문, J11 논점 reason, J12 비교, R19 입력 경계를 구현한다.
- 작업 5: 카탈로그의 허용 source·section 안에서만 특약 RAG를 수행한다.
- 작업 6~10: 생성·저장·화면·오프라인 평가에서 이 경계를 회귀검증한다.
- 작업 11: 독립 검토와 실제 provider 검증 전까지 데이터 상태를 `unverified`·`draft_pending_human_review`로 유지한다.
