# 파인튜닝 데이터셋 구조 (로컬 7B)

> 로컬 7B 모델의 조항 분류 파인튜닝 데이터셋 정의. 데이터는 [`data/datasets/`](../../data/datasets/), 라벨 체계는 [`data/labels/`](../../data/labels/).
> 로컬 7B는 **조항 유형·명확성·책임 주체·조건·확인 필요 후보를 분류**한다. **최종 판정은 내리지 않는다**(최종 판정은 규칙 엔진, [`judgment-spec.md`](judgment-spec.md)).

## 입력·출력

- **입력(input)**: 계약서·특약의 **조항 문장 1건**(또는 짧은 조항 단위 텍스트).
- **출력(label)**: 아래 5종 라벨.

| 라벨 | 의미 | 값 예시(초안) |
|------|------|----------------|
| `clause_type` | 조항 유형 | 보증금 / 차임 / 관리비 / 계약기간 / 원상복구 / 반환조건 / 특약 기타 |
| `clarity` | 명확성 | 명확 / 불명확 |
| `responsible_party` | 책임 주체 | 임대인 / 임차인 / 공동 / 불명확 / 해당없음 |
| `condition` | 조건·기한 명시 여부 | 명시 / 미명시 |
| `review_required` | 확인 필요 후보 | true / false |

- 라벨 값 목록은 초안이며, 라벨 정의는 [`data/labels/`](../../data/labels/)에서 확정한다.
- 로컬 7B 출력은 규칙 엔진의 **입력 보조**로만 쓰인다. `clarity`·`review_required`가 판정 상태를 직접 확정하지 않는다.

## 데이터 흐름

```
datasets/source/     원천 문장 (비식별·합성, 라벨 없음)
      ↓ 라벨링·검수
datasets/labeled/    라벨 부착·검수 완료 (train/val/test 분할 전)
      ↓ 분할 (문장 중복 없이)
datasets/train/      학습
datasets/validation/ 하이퍼파라미터·조기종료
datasets/test/       최종 평가 (학습·검증에 미사용)
```

## 분리 원칙

- **train / validation / test를 분리한다.** 동일 조항 문장이 여러 분할에 중복되지 않게 한다(누수 방지).
- test는 최종 성능 측정 전까지 열지 않는다.
- 분할 비율·시드·건수는 데이터 확보 후 확정(TODO).

## 라벨링 기준

- 조항 문장 1건 = 라벨 5종 1세트.
- 판단 모호 문장은 `clarity=불명확` 또는 `condition=미명시`로 표기하고, 라벨러가 임의로 단정하지 않는다.
- 라벨 정의·가이드·경계 사례는 [`data/labels/`](../../data/labels/) 각 폴더에 문서화한다.

## 비식별·합성 원칙

- 실제 개인정보를 포함하지 않는다. 실명·실주소·실계좌를 가상값으로 치환하거나 합성 생성한다.
- 실제 계약서에서 파생한 문장은 재식별 불가하게 비식별한 뒤에만 사용한다.

## 검수 상태

- 각 데이터 항목에 검수 상태(`draft` / `reviewed`)를 기록한다.
- `reviewed`만 train/validation/test 분할 대상으로 승격한다.
