# 데이터 흐름

로그인 사용자의 **계약 건(ContractProject)** 단위로 데이터가 이동·저장·재조회된다. 단일 세션이 아니라 계약 건에 묶여 영속된다.

## 계약 건 단위 흐름

```
업로드 → 추출 → 확인·수정 → 분석 → 판정·근거 → 저장 → 재조회
```

각 단계 데이터는 계약 건 식별자에 묶여 저장되고, 재로그인·재방문 시 계약 건 단위로 다시 불러온다.

## 1. 입력 데이터 (업로드)

- **필수**: 계약서·특약
- **선택**: 등기사항증명서, 중개대상물 확인설명서
- 실제 계약 문서·개인정보는 Git 제외. 비식별·합성 샘플만 `data/sample/`.

## 2. 처리 데이터 (추출 → 확인·수정 → 분석)

- 추출값 (`data/processed/extracted-fields/`) — 문서 인식·추출 결과
- 정규화 문서 (`data/processed/normalized-documents/`)
- **사용자 확인·수정본** — 분석은 원 추출값이 아니라 사용자가 확인·수정한 값을 입력으로 사용한다
- 상용 LLM 조항 구조화·불명확성 후보(Gemini 3.5 Flash) — (선택)로컬 7B 성능비교 실험은 MVP 크리티컬 패스 제외
- RAG 청크 (`data/rag/chunks/`)

## 3. 결과 데이터 (판정·근거)

- 규칙 엔진 판정 결과 (판정 상태 9개, 시급도 5개 — 정의는 [`../../AGENTS.md`](../../AGENTS.md))
- 원문 증거(문서 내 근거 위치)
- 공식 근거(RAG 출처)
- 임대인·공인중개사 대상 질문
- 서명 전 체크리스트
- 계약 직후 권리 확보 행동
- 미완료·확인 필요 항목
- 결과 리포트

## 4. 저장·재조회

- 회원별 계약 건에 묶어 문서 메타데이터·추출값·판정·근거·질문·행동·리포트를 저장한다.
- 체크리스트 상태·계약 직후 행동 상태를 저장하고 재방문 시 이어서 관리한다.
- 추출·판정·근거 오류에 대한 사용자 피드백을 저장한다.
- 저장·재조회 API·인증 세부는 `auth-and-persistence`(예정)에서 정의한다.

## 원칙

- 원본을 직접 덮어쓰지 않는다. 추출·정규화·확인·수정본은 별도 산출물로 남긴다.
- 실제 자료와 생성 데이터를 명확히 구분한다.
- 실제 계약 문서·개인정보·모델 가중치·체크포인트·운영 결과물은 Git에서 제외한다. 비식별·합성 데이터만 커밋한다.
- 종합 판정(`안전`/`위험`/점수)을 산출하거나 저장하지 않는다.
