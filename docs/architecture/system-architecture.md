# 시스템 아키텍처

회원 기반 모바일 웹앱에서 사용자가 **계약 건(ContractProject)** 과 **계약 문서**를 관리하고, AI 파이프라인이 조항 분류·문서 교차검증·공식 근거·쉬운 설명을 제공한다. 분석은 단일 세션이 아니라 **로그인 사용자의 계약 건 단위**로 저장·재조회한다.

## 계층 구성

```
[ frontend ]  모바일 웹앱 — 회원·계약 대시보드·업로드·추출값 확인/수정·리포트·체크리스트
      │  HTTP (REST)
      ▼
[ backend ]   FastAPI — 회원·계약 건·문서·분석 실행·결과·체크리스트·피드백 API
      │        파이프라인 오케스트레이션 + 영속 저장
      │  함수 호출 (ai/ 패키지 인터페이스)
      ▼
[ ai ]        문서 인식(PDF·OCR·VLM) · 추출 · 정규화 · 상용 LLM 조항 구조화(Gemini 3.5 Flash)
              · 규칙 엔진 · RAG · 상용 LLM 재검토·생성 · guardrail · routing · (선택)로컬 7B 성능비교 실험(MVP 크리티컬 패스 제외)
      │  참조
      ▼
[ data ]      비식별·합성 샘플 · 스키마 · 규칙 · 라벨 · 파인튜닝 데이터셋
              · RAG 자료·메타데이터 · 평가 데이터 · 모델 메타데이터
```

`data`는 저장 계층이 아니라 `ai`를 뒷받침하는 기준 데이터·자산이다. 사용자별 계약 건·분석 결과의 영속 저장은 `backend`가 담당한다.

## 책임 경계

- **frontend → backend**: REST 요청. 회원 인증, 계약 건 생성·조회·삭제, 문서 업로드, 추출값 확인·수정, 분석 요청, 리포트·체크리스트·행동 상태 조회.
- **backend → ai**: 백엔드는 오케스트레이션과 저장만 담당한다. AI 로직을 중복 구현하지 않고 `ai/` 파이프라인 인터페이스를 호출한다.
- **ai → data**: 분석 기준 데이터(스키마·규칙·RAG 자료·모델 메타데이터)와 평가 데이터를 참조한다. 사용자 데이터를 직접 저장하지 않는다.
- **backend ↔ 영속 저장소**: 회원, 계약 건, 문서 메타데이터, 추출값(확인·수정본), 판정 결과, 근거, 질문·행동, 체크리스트·행동 상태, 피드백을 저장·재조회한다.

## 컴포넌트 경계 (책임 분리)

| 컴포넌트 | 책임 | 하지 않는 것 |
|----------|------|--------------|
| 문서 인식(PDF·OCR·VLM) | 문서에서 텍스트·레이아웃 획득 | 판정·분류 |
| 추출 | 핵심 필드 구조화 추출 | 최종 판정 |
| 로컬 7B 모델 (선택적 실험, MVP 크리티컬 패스 제외) | 상용 vs 로컬 성능비교 실험용 조항 분류(파인튜닝 B/C안). MVP 조항 구조화는 상용 LLM(Gemini 3.5 Flash)이 담당 | **최종 판정**, 규칙 결과 변경 |
| Python 규칙 엔진 | 문서 내부 판정 + 문서 교차검증의 **명시적 최종 판정** | 근거 생성, 설명 |
| RAG | 공식 자료 기반 **근거 검색** | **판정** (근거 없으면 `확인 불가`/`확인 필요` 반환) |
| 상용 LLM | 저신뢰 결과 재검토, 근거 기반 쉬운 설명·질문·행동 생성 | 규칙 판정 변경 |
| guardrails | 단정 표현·근거 없는 출력 차단 | — |
| routing | 단계별 처리 모델·fallback 선택 | 판정 |
| backend | 전체 오케스트레이션·저장 | AI 로직 재구현 |

파이프라인 단계별 흐름은 [`ai-pipeline.md`](ai-pipeline.md), 데이터 이동은 [`data-flow.md`](data-flow.md) 참조.

## 계약 건 단위 저장 모델

- 최상위 저장 단위는 **회원(User)** 과 회원이 소유한 **계약 건(ContractProject)** 이다.
- 계약 건 하나에 여러 계약 문서, 추출값, 분석 결과, 리포트, 체크리스트·행동 상태가 묶인다.
- 분석 결과는 종료되지 않고 저장되어, 재로그인·재방문 시 계약 건 단위로 재조회·상태 관리된다.
- 결과 표현은 **판정 상태 9개**와 **시급도 5개**(정의는 [`../../AGENTS.md`](../../AGENTS.md), [`../data/judgment-spec.md`](../data/judgment-spec.md))를 따르며 `안전`/`위험`/점수 같은 **종합 판정을 사용하지 않는다.**

## 로컬 모델 배치 원칙

로컬 7B는 MVP 크리티컬 패스에서 제외되며, 상용 vs 로컬 성능비교를 위한 **선택적 병렬 실험(파인튜닝 B/C안)**으로만 유지한다. 별도 모델 서버 배포가 확정되지 않았으므로 `model-service`·`inference-server` 같은 최상위 서비스를 임의로 추가하지 않는다. 실험을 진행하는 경우 로컬 7B 기능은 우선 `ai/` 내부 인터페이스로 분리해 결합도를 낮추고, 추후 별도 서빙이 결정되면 `services/model-api`로 분리할 수 있게 한다. 배포 관점은 [`deployment.md`](deployment.md) 참조.
