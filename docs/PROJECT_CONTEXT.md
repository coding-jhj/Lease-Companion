## 1. 서비스 정의

### 1.1 목적

슬기로운 계약생활은 첫 전월세 계약을 준비하거나 계약 경험이 적은 **2030 청년 임차인**을 위한 계약 전 확인 지원 서비스다.

서비스는 계약서와 관련 데이터를 구조화하고 교차검증하여, 사용자가 계약 전에 놓치기 쉬운 **위험 신호**를 발견하도록 돕는다. 여기서 위험 신호란 사기·위법·계약 안전 여부의 결론이 아니라, 사용자가 추가로 확인해야 하는 불일치·누락·불명확성·권리 기재·검증 불가 상태를 뜻한다.

핵심 포지셔닝은 다음과 같다.

> 전세사기로 이어질 수 있는 위험 신호를 계약 전에 인지하게 하고, 문제 발생 시 대응할 수 있도록 확인 행동과 증빙 준비를 돕는다.

### 1.2 서비스가 제공하는 가치

- 계약 문서에서 확인해야 할 정보와 조항을 구조화한다.
- 계약서와 관련 데이터 사이의 불일치·누락·불명확성을 규칙으로 검사한다.
- 각 결과에 원문 증거와 공식 근거를 연결한다.
- 결과를 임대인·공인중개사에게 물을 질문으로 변환한다.
- 서명 전 체크리스트와 계약 직후 행동을 제공한다.
- 계약 과정에서 확인한 내용과 증빙을 다시 찾을 수 있는 임차권 방어 리포트로 정리한다.

### 1.3 대상과 확장 범위

- **MVP 주 사용자**: 주거용 전세·보증부 월세·일반 월세 계약을 준비하는 2030 청년 임차인
- **향후 확장 사용자**: 공인중개사, 주거상담기관
- **제외 범위**: 부동산 매매, 상가·사무실 임대차, 경매·공매, 소송, 계약 종료 후 보증금 반환 분쟁, 거주 중 하자 대응

## 2. 서비스 핵심 원칙

1. 전세사기 여부, 계약 안전 여부, 계약 가능 여부, 적법·위법 여부를 단정하지 않는다.
2. `안전`, `사기`, `계약 가능`, `위험 점수`, `사기 가능성 점수` 같은 종합 법률 판단을 생성하지 않는다.
3. 결과는 규칙별 `일치`, `불일치`, `명확`, `불명확`, `미기재`, `상충 가능`, `확인 필요`, `확인 불가`, `적용 제외` 중 해당 규칙에 허용된 상태로 표현한다.
4. 시급도는 사기 확률이나 계약 안전도가 아니라 **사용자가 언제 확인·행동해야 하는지**를 나타낸다.
5. 확인할 수 없는 값은 추측하거나 보완 생성하지 않고 `확인 불가` 또는 해당 규칙의 데이터 부족 상태로 처리한다.
6. LLM은 정보를 구조화하고 설명을 생성하지만 최종 규칙 판정을 임의로 내리거나 바꾸지 않는다.
7. Python 규칙 엔진의 구조화된 결과를 판정 기준으로 우선한다.
8. RAG는 법령·표준계약서·공공기관 안내 등 공식 자료를 설명의 근거로 검색하며 판정하지 않는다.
9. 사용자는 분석 전에 AI 추출값을 원문과 대조하고 직접 수정할 수 있어야 한다.
10. 판단 결과에는 근거와 함께 질문, 권장 행동, 체크리스트, 보관할 증빙을 제공한다.
11. 추출값, 사용자 수정값, 규칙 판정, LLM 생성값을 서로 구분한다.
12. 규칙 버전·프롬프트 버전·분석 이력을 보존하여 같은 입력과 버전으로 결과를 재현할 수 있어야 한다.

## 3. 사용자 흐름

아래 12단계는 최종 서비스 경험을 상세화한 흐름이다. 기존 8단계 기준과의 관계는 다음과 같다.

| 기존 8단계 | 상세 서비스 흐름 |
|---|---|
| 1. 회원가입·로그인 | 계약 건에 결과를 저장·재조회하기 위한 사용자 진입 |
| 2. 계약 대시보드·계약 건 생성 | 분석 대상 계약 건 생성 |
| 3. 계약 상황 입력 | 아래 2단계 |
| 4. 문서 업로드 | 아래 1단계 및 관련 데이터 연결 |
| 5. 추출값 확인·수정 | 아래 3~5단계 |
| 6. 분석 | 아래 6~9단계 |
| 7. 리포트 | 아래 8~12단계 결과 표시 |
| 8. 상태 관리 | 체크리스트·계약 직후 행동·증빙 상태 저장 및 재조회 |

### 상세 12단계

1. 사용자가 계약 건을 만들고 계약서 PDF를 업로드한다.
2. 사용자가 계약 유형·계약 단계 등 현재 계약 상황을 입력한다.
3. 문서 인식 계층이 디지털 PDF 텍스트를 추출하고, 스캔 문서는 확정된 VLM 경로로 인식한다.
4. 상용 LLM이 버전이 명시된 JSON Schema에 맞춰 계약 정보를 구조화한다.
5. 사용자가 원문과 추출 결과를 대조하고 필요한 값을 수정한 뒤 확인을 완료한다.
6. 사용자 확인값과 관련 데이터를 핵심 Python 규칙 엔진으로 검사한다.
7. 법령·표준계약서·공공기관 자료를 검색하여 RAG 기반 설명 근거를 연결한다.
8. 규칙별 결과 상태와 행동 시급도를 표시한다.
9. 임대인·공인중개사에게 물어볼 확인 질문을 생성한다.
10. 서명 전 체크리스트를 제공한다.
11. 계약 직후 필요한 권리 확보 행동과 보관할 증빙을 안내한다.
12. 원문 증거·공식 근거·질문·행동·체크리스트를 임차권 방어 리포트로 생성하고 계약 건에 저장한다.

R01~R10은 현재 최소 구현 슬라이스의 규칙 식별자다. 기존 전체 MVP의 J01~J12 판정 체계와 동일한 축으로 간주하거나 번호를 치환하지 않는다. 전체 MVP 범위에 대한 최종 관계는 아래 충돌 항목의 결정을 따른다.

근저당권·압류·가압류·신탁등기의 존재만으로 계약의 안전·위험·적법 여부를 판정하지 않는다. 해당 기재의 의미와 계약 권한·선순위 관계를 확인할 질문과 행동으로 연결한다.

### 4.2 규칙 결과 계약

각 규칙 결과는 최소한 다음 정보를 구조적으로 제공하는 것을 목표로 한다.

| 필드 | 의미 |
|---|---|
| `rule_id` | 버전 관리되는 규칙 식별자 |
| `rule_name` | 사용자에게 표시할 규칙명 |
| `status` | 해당 규칙에 허용된 결과 상태 |
| `urgency` | 행동 시점·확인 우선순위 |
| `summary` | 무엇이 확인되었거나 부족한지에 대한 쉬운 요약 |
| `evidence` | 판정에 사용한 원문 인용 또는 구조화된 증거 |
| `source_document` | 증거가 나온 문서 식별자와 문서 유형 |
| `page` | 원문 증거 페이지. 확보 불가하면 추측하지 않고 사유 기록 |
| `question` | 임대인·공인중개사 등에게 물을 질문 |
| `recommended_action` | 사용자가 수행할 확인·요청·보관 행동 |
| `unverifiable_reason` | `확인 불가`인 이유. 확인 가능하면 `null` |

추가로 재현성과 추적성을 위해 `rule_version`, `analysis_run_id`, `input_snapshot_version`, `created_at`을 저장 모델에서 함께 관리한다. 구체 스키마는 API·DB 설계 시 확정한다.

## 5. 시급도와 화면 표현

### 5.1 의미

- 시급도는 사기 확률이나 계약 안전도를 나타내지 않는다.
- 색상만으로 상태를 전달하지 않고 텍스트 라벨·아이콘 등 접근 가능한 표현을 함께 사용한다.
- 판정 상태와 시급도는 독립된 필드다.

### 5.2 제안된 사용자 화면 그룹

- **빨간색 — 반드시 확인**
- **노란색 — 확인 권장**
- **초록색 — 일반 확인**

저장·API는 기존 공식 시급도 5개를 유지하고 사용자 화면에서만 다음과 같이 묶는다: `즉시 확인`·`분석 불가` → **반드시 확인**, `계약 전 확인`·`계약 직후 조치` → **확인 권장**, `참고` → **일반 확인**. 화면 그룹은 내부 시급도를 대체하지 않으며 문구·아이콘을 색상과 함께 제공한다.

## 6. 시스템 아키텍처

```text
사용자 모바일 웹앱
  ├─ 회원·계약 건 관리
  ├─ 계약서 PDF 업로드·계약 상황 입력
  ├─ 추출값 확인·수정
  └─ 확인 리포트·체크리스트·행동 상태
          │
          ▼
FastAPI 백엔드
  ├─ 인증·사용자 API
  ├─ 계약 건·문서 처리 API
  ├─ 추출 확인 API
  ├─ 분석·리포트 API
  ├─ 체크리스트·피드백 API
  └─ AI 파이프라인 오케스트레이션·영속 저장
          │
          ▼
AI 문서 처리·분석 파이프라인
  문서 인식 → JSON Schema 추출 → 사용자 확인값
  → Python 규칙 엔진 → 공식자료 RAG → 설명·질문·행동 생성
  → Guardrail → 버전이 기록된 분석 결과
          │
          ▼
데이터·지식 저장 계층
  원본 문서 저장소 / 계약 정보 DB / 모의·비식별 관련 데이터
  / 법령·공식자료 검색 인덱스 / 분석 결과·이력 저장소
```

### 6.1 사용자 화면

- 회원가입·로그인 및 계약 대시보드
- 계약서 PDF 업로드
- 계약 상황 입력
- 추출값과 원문 근거 확인·수정
- 규칙별 상태·시급도·원문 증거·공식 근거 조회
- 질문·서명 전 체크리스트·계약 직후 행동 관리
- 임차권 방어 리포트 저장·재조회

프론트엔드는 React + TypeScript + Vite 기반 SPA로 구현한다. 라우팅은 React Router, API mock은 MSW, 단위·컴포넌트 테스트는 Vitest + Testing Library, 패키지 관리는 npm을 사용한다. 상세 결정은 [`decisions/2026-07-16-frontend-react-vite.md`](decisions/2026-07-16-frontend-react-vite.md)를 따른다.

### 6.2 백엔드 서비스

- **문서 처리 API**: 업로드 검증, 문서 식별, 인식·추출 실행, 원문/추출 버전 관리
- **계약 건 관리 API**: 사용자별 계약 건, 계약 상황, 문서, 체크리스트 상태 관리
- **분석·리포트 API**: 사용자 확인 완료 검증, 분석 실행·상태·결과·이력·피드백 관리
- **오케스트레이션**: AI 패키지 호출, 오류·재시도·fallback 처리, 저장 경계 유지

백엔드는 AI 추출·규칙·RAG 로직을 중복 구현하지 않는다. 데이터베이스는 PostgreSQL, 인증은 JWT Bearer(PyJWT) + Passlib-bcrypt로 확정했다(2026-07-16). Access Token 만료는 로컬 MVP 24시간 임시값이며 refresh token·토큰 폐기·운영 서명 키 관리와 운영 배포 플랫폼은 미정(TODO)이다.

### 6.3 AI 문서 처리 파이프라인

1. 파일 형식·크기·페이지·문서 유형을 검증한다.
2. 디지털 PDF는 텍스트 레이어를 직접 추출한다.
3. 스캔·이미지 문서는 확정된 Gemini VLM 통합 경로로 텍스트와 구조를 인식한다. 별도 OCR 서비스 추가 여부는 결정 문서 없이는 변경하지 않는다.
4. 추출 결과를 버전이 명시된 JSON Schema로 검증한다.
5. 사용자가 확인·수정한 값을 분석 입력 스냅샷으로 고정한다.
6. Python 규칙 엔진이 문서 내부 검사와 관련 데이터 교차검증을 수행한다.
7. RAG가 법령·표준계약서·공공기관 안내에서 근거를 검색한다.
8. 상용 LLM이 규칙 결과와 검색 근거 범위 안에서 쉬운 설명·질문·체크리스트·행동을 생성한다.
9. Guardrail이 단정 표현, 근거 없는 행동, 규칙 판정 변경을 차단한다.
10. backend가 입력·규칙·프롬프트·근거 버전과 결과를 저장한다.

### 6.4 데이터 및 지식 저장소

- **원본 문서 저장소**: 업로드 계약서와 문서 메타데이터. 제품·보존 기간·암호화·삭제 정책은 TODO
- **계약 정보 DB**: PostgreSQL. 계약 건, 계약 상황, 추출값, 사용자 수정값, 체크리스트 상태
- **관련 문서 데이터**: 모의·비식별 등기사항증명서 등 규칙 교차검증용 데이터
- **법령·공식 자료 검색 저장소**: Chroma 로컬 모드. 공식 원문, 청크, 출처 메타데이터, 임베딩/검색 인덱스
- **분석 결과 및 이력 저장소**: 입력 스냅샷, 규칙·프롬프트·모델·근거 버전, 결과, 피드백

### 6.5 사용자 제공 결과

- 규칙별 상태와 위험 신호 설명
- 행동 시급도
- 원문 증거와 페이지
- 법령·공식 자료 근거
- 임대인·공인중개사에게 물어볼 질문
- 서명 전 체크리스트
- 계약 직후 권리 확보 행동
- 계약서·이체내역·대화 기록 등 보관할 증빙
- 임차권 방어 리포트

## 7. 데이터 처리 원칙

1. 계약서 PDF는 사용자가 업로드한다.
2. 등기사항증명서는 사용자 업로드와 합성 CASE용 모의 등기 연결(`registry-link`)을 모두 지원한다.
3. OCR·문서 인식 원문, LLM 최초 추출값, 사용자 수정값을 서로 다른 데이터로 저장한다.
4. 규칙 검사는 반드시 사용자가 확인 완료한 값을 우선 사용한다.
5. 분석 결과에는 가능한 경우 원문 인용, 문서 식별자, 페이지를 포함한다.
6. 원문 위치를 확보하지 못하면 추측하지 않고 `source_evidence.page`·`text`를 null로 유지하며 판독 실패는 `failure_reason`에 기록한다.
7. 규칙 버전, 입력 스냅샷, 프롬프트·모델 버전, 검색 근거 버전, 분석 시각을 저장한다.
8. 실제 계약 문서와 개인정보는 Git에 커밋하지 않는다. 저장소의 샘플·평가 데이터는 합성 또는 완전 비식별 데이터만 사용한다.
9. 로그와 오류 메시지에 개인정보나 문서 원문을 남기지 않는다.
10. 외부 모델 API로 원문 이미지·텍스트를 보내는 범위와 고지·동의·마스킹 정책은 프로덕션 전에 확정한다.

## 8. 구현 우선순위

1. 계약서 PDF 입력
2. 디지털 PDF 텍스트 추출과 스캔 문서 인식
3. JSON Schema 기반 계약 정보 추출
4. 사용자 추출값 확인·수정
5. 핵심 규칙 10개
6. 공식 자료 기반 RAG 설명
7. 질문 및 서명 전 체크리스트 생성
8. 계약 직후 행동 안내
9. 임차권 방어 리포트 생성·저장·재조회

이 우선순위는 구현 순서를 나타내며, 기존 PoC·최소 MVP·전체 MVP 범위를 혼용하지 않는다. 각 단계 구현 전 미정 기술과 아래 충돌 항목을 확인한다.

## 9. 구현 원칙

- 이번 문서 정비는 기능 구현을 포함하지 않는다.
- 컴포넌트 책임을 유지한다: 인식·추출은 AI 계층, 판정은 규칙 엔진, 근거 검색은 RAG, 오케스트레이션·저장은 backend, 사용자 상호작용은 frontend가 담당한다.
- LLM 자유 텍스트를 내부 판정 계약으로 직접 사용하지 않고 JSON Schema로 검증한다.
- 원본 추출값을 사용자 수정값으로 덮어쓰지 않는다.
- 규칙 판정을 생성 LLM이 변경할 수 없도록 입출력 경계를 분리한다.
- 공식 근거가 없으면 출처를 만들거나 행동을 확정하지 않는다.
- 프롬프트·규칙·스키마를 버전 관리한다.
- 개인정보 최소수집, 접근 통제, 암호화, 보존·삭제 정책을 제품·배포 설계에 포함한다.
- 모바일 우선 접근성과 로딩·오류·빈 데이터·분석 중 상태를 설계한다.
- 측정하지 않은 정확도·성능·비용 목표를 임의로 작성하지 않는다.
- 미정 기술은 `TODO`, 기존 기준과의 충돌은 결정 대기 항목으로 남긴다.
