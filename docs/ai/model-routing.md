# 모델 라우팅 · 상용/로컬 운영 구조

`routing` 모듈이 단계별 처리 모델과 fallback을 선택하는 규칙을 정의한다. 7.13 피드백 준비사항 #9 "상용 LLM과 로컬 모델의 운영 구조" 산출물이며, 2026-07-14 팀 확정 최종 모델·기술 선정표(13단계)를 반영한다.

## 원칙 (확정)

- **무거운 추론은 상용 API, 개인정보·최종 판정은 전부 로컬.** 상용 LLM을 우선 사용하되(PDF §2.3 1안), 개인정보는 API로 나가지 않는다. **단 OCR 단계는 예외 — 아래 데모 예외 참조.**
- **개인정보 비식별 게이트.** 상용 API 호출(구조화·검색·생성) 전에 로컬에서 PII를 토큰화하고, API는 비식별 텍스트만 수신한다. (privacy-by-design, 아래 별도 절)

> ⚠️ **데모 단계 예외 (2026-07-14 결정).** OCR이 로컬 PaddleOCR-VL → 상용 Gemini VLM으로 바뀌면서, 스캔·사진 입력의 **원문 이미지가 비식별 전에 API로 전송**된다. 즉 위 두 원칙은 스캔·사진 경로에서 현재 성립하지 않는다. 디지털 PDF 경로(PyMuPDF 로컬 추출)는 원칙대로 유지된다. 데모 단계 수용, **프로덕션 전 재검토 필수**(이미지 마스킹 또는 로컬 경량 OCR). → [`../decisions/2026-07-14-ocr-gemini-integration.md`](../decisions/2026-07-14-ocr-gemini-integration.md)
- **할당량 초과·장애 시 fallback.** 생성 API 호출 실패·할당량 소진·시간 초과 시 provider 제한 재시도 후 결정적 안전 템플릿으로 저하한다. 미구현 대체 모델을 호출하지 않는다.
- **규칙 엔진 판정은 라우팅 대상이 아니다.** 결정론적으로 항상 로컬에서 수행하며, 어떤 모델도 규칙 결과를 변경하지 않는다.

## 단계별 처리 모델 선택

선정표 13단계를 라우팅 관점(로컬 vs 상용 API + fallback)으로 정리한 것이다.

| 처리 | 위치 | 모델/기술 | fallback |
|------|------|-----------|----------|
| 디지털 PDF 추출 | 로컬 | PyMuPDF · PDF.js | 텍스트 레이어 없음 → OCR |
| 스캔·사진 OCR | **API** ⚠️ | Gemini 3.5 Flash (VLM 통합) | 503 백오프 재시도 → 실패 시 사용자 재촬영·수동 입력 |
| PII 탐지·치환 | 로컬 | privacy-filter-korean + 정규식 → 결정론 토큰화·매핑표 | 형식형은 정규식이 항상 커버 |
| 필드 구조화 | **API** | Gemini 3.5 Flash | 설정 부재·오류·할당량·응답 검증 실패 시 로컬 정규식 추출 |
| 추출값 확인·수정 | 로컬 웹 | 사용자 UI (필수) | 없음 |
| 위험 판정 (최종) | 로컬 | Python 규칙 엔진 (결정론) | 없음 (항상 수행) |
| 공식 근거 검색 | API+로컬 | gemini-embedding-001 + BM25 (Top-20) | embedding·Chroma 실패 시 BM25 |
| 공식 근거 재정렬 | **API** | Cohere rerank-v4.0-pro (Top-20→Top-5) | 실패 시 hybrid 순위 유지 |
| 설명·질문·행동 생성 | **API** | Gemini 3.5 Flash (`gemini-3.5-flash`) | provider 제한 재시도 후 안전 템플릿 |
| 복원·표시·삭제 | 로컬 | 매핑표 복원 → 세션 종료 시 전량 삭제 | 없음 |

## 개인정보 비식별 게이트 (privacy-by-design)

상용 API를 우선 쓰면서 개인정보 유출을 차단하는 핵심 구조. (상세: [`../data/privacy-policy.md`](../data/privacy-policy.md), 차별성: [`../planning/differentiation.md`](../planning/differentiation.md))

1. 로컬에서 PII 탐지(문맥형 + 형식형) → 결정론적 토큰화(홍길동→`[PERSON_1]`, 주소→`[ADDRESS_1]`), 매핑표는 로컬 보관.
2. 상용 API 단계(구조화·검색·생성)는 **비식별 텍스트만** 수신.
3. 결과는 로컬 매핑표로 복원, 실제값은 필요한 화면에서만 표시.
4. 세션 종료 시 원문·OCR 임시파일·토큰 매핑표를 전량 삭제.

## 최소 routing 실행 계층 (2026-07-18)

- `routing/models.py`가 단계·primary·선택 대상·사용 가능 여부·fallback 여부·실패 사유를 고정한다.
- `routing/service.py`가 provider를 한 번 실행하고 정해진 로컬 저하 경로를 선택한다.
- 실패 사유는 provider 미사용 가능·일반 오류·할당량 초과·응답 검증 실패로 구분한다.
- 결정은 `ai_routing_decision` 구조화 로그와 추출·검색 실행 결과에 남긴다. 민감 입력과 provider 원문 예외는 기록하지 않는다.
- 미구현 CLI 모델이나 로컬 7B를 fallback으로 호출하지 않는다.

## 로컬 모델 경계

- **MVP 크리티컬 패스의 로컬 실행 = 모델 없음(2026-07-14 OCR 변경 이후).** OCR까지 상용 API로 넘어가면서 로컬에서 도는 것은 결정론적 코드뿐이다 — PyMuPDF 텍스트 추출, PII 토큰화, Python 규칙 엔진, BM25. GPU가 필요한 모델 추론은 크리티컬 패스에 하나도 없어 Colab 환경 OOM·중단 위험이 사라졌다.
- **반대급부: 원문 이미지 PII 노출.** 로컬 OCR을 없앤 대가로 스캔·사진 경로의 비식별 게이트가 뚫렸다(위 데모 예외). 로컬 경량 OCR 복귀는 이 트레이드오프의 되돌림 경로다 — 후보: [`local-model-plan.md`](local-model-plan.md).
- **로컬 7B는 MVP 크리티컬 패스에서 제외됨(선정표).** "상용 vs 로컬 성능 비교"(PDF §4.4, 파인튜닝 B안) 병렬 실험으로만 선택적으로 검토한다. 규칙 엔진 판정을 대체하지 않는다.

## 관측 지표 (측정 후 기록)

단계별 상용 API 호출 성공률·fallback 발생률, 문서당 상용 API 비용, 문서당 처리 시간, 근거 인용율. 정의·기록 형식: [`evaluation-matrix.md`](evaluation-matrix.md). 목표 수치는 측정 전 만들지 않는다.
