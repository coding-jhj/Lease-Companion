## 2. 주요 논의 내용

### 2.1 서비스 구현 형태
- 계약 현장에서 촬영·업로드하는 형태라 **모바일 환경이 중요**. PC 중심 웹은 실제 사용자 관점과 불일치.
- 안드로이드 네이티브 앱은 인력·기간 부족. 단순 반응형 웹만으론 서비스 특성을 살리기 어려움.
- **결론**: MVP는 **모바일 최적화 웹앱**으로 개발. 스마트폰에서 업로드·결과 확인·질문 카드 확인 가능하게 설계. 네이티브 앱은 확장 단계에서 검토.

### 2.2 현재 기획안의 핵심 문제
- 계약서 요약·정보 추출·확인 항목·질문·체크리스트 생성은 **대부분 범용 LLM으로 구현 가능** → 기술적 차별성 약함.
- 부족한 검증 기능: 취약 조항 판단 / 사기 위험 신호 식별 / 공공 API 연동 실제 정보 검증 / 자체 데이터·모델 차별성.
- **핵심 시사점** — 차별성 확보 위해 다음 중 최소 하나가 명확해야 함:
  - 자체 계약·법률 특화 모델
  - 계약 데이터셋 파인튜닝
  - 법령·공공자료 기반 RAG
  - 문서 간 정보 교차 검증
  - 공공 API 실제 정보 검증
  - 범용 LLM 대비 성능 비교·평가

### 2.3 LLM 모델 선정
- 계약·법률 문서는 소형 모델엔 난도 높음. 2B급은 부족 가능성. 로컬은 최소 7B 양자화 검토.
- 7B도 데이터 구축·파인튜닝·추론 환경이 필요해 1명 단독 부담. 로컬 GPU·Colab 안정성도 고려.
- **1안(상용 우선)**: Gemini·GPT 등 검증된 상용 LLM 1순위, 토큰·할당량 소진 시 Claude CLI·Codex CLI 등으로 분기, 다중 모델 구조.
- **2안(로컬)**: 7B 양자화 실행, 계약·법률 파인튜닝, 상용 대비 성능 평가.
- **결론**: 안정적 구현 위해 **상용 LLM 우선**이 현실적. 단, 단순 API 호출로 보이지 않도록 RAG·데이터 검증·성능 평가 추가. 로컬·파인튜닝은 학습 목표·GPU 지원 고려해 추가 검토.

### 2.4 OCR 및 VLM 적용
- 텍스트 레이어 없는 스캔 PDF·촬영 이미지 존재 → OCR 완전 배제 불가. 표·도장·체크박스·배치 분석엔 OCR만으론 부족, VLM 보조 필요할 수 있음. (후보로 Qwen 2.5 계열 7B VLM 언급)
- **결론(문서 입력 처리 구조)**:
  1. PDF/이미지 업로드
  2. 텍스트 레이어 존재 여부 확인
  3. 있으면 직접 텍스트 추출
  4. 없으면 OCR 수행
  5. 표·레이아웃·체크박스 필요 시 VLM 보조
  6. 추출 결과 정규화 → LLM·규칙 엔진 전달

### 2.5 RAG 데이터 구성
- 단순 법령 벡터 저장·검색만으론 차별성 부족(상용 LLM이 일반 법령을 이미 앎) → RAG 적용 전후 차이 검증 필요. 어떤 데이터·질문에 쓸지 미정의.
- **RAG 후보 데이터**: 주택임대차보호법·시행령 / 전세사기 예방 공공 가이드 / 국토부 표준 임대차계약서 / 중개대상물 확인·설명서 작성 기준 / 전세사기·보증금 반환 분쟁 사례 / 등기사항증명서 항목 해설 / 건축물대장 항목 해설 / 계약 특약 사례·위험 문구 / 전입신고·확정일자·임대차 신고 절차 / HUG 전세보증 가입 조건·안내.
- **검증 방안**: 범용 LLM 단독 vs RAG 적용 비교 (법령 근거 정확성 / 근거 인용 여부 / 잘못된 법률 정보 생성 여부 / 질문·체크리스트 구체성).
- **결론**: RAG 데이터 조사·목록 확정 우선. RAG는 단순 검색이 아니라 **근거 제공·환각 감소** 용도. 범용 vs RAG 성능 비교를 결과에 포함 검토.

### 2.6 파인튜닝 적용 여부
- 파인튜닝은 데이터 수집·정제·라벨링·학습·평가에 상당 시간. 3명이 서비스+모델 병행 부담. 적은 데이터로 무리한 파인튜닝은 오히려 성능 저하 위험.
- **방안 비교** (우선순위 A → B → C):

| 방안 | 구성 | 특징 |
|------|------|------|
| A안 | 상용 LLM + RAG | 가장 현실적, MVP 적합 |
| B안 | 로컬 7B + RAG | GPU·모델 학습 경험 확보 |
| C안 | 로컬 7B 파인튜닝 + RAG | 차별성 높으나 개발 난도 큼 |

### 2.7 역할 분담
- **현재 문제**: 모델 담당 미정. AI·OCR·VLM·RAG를 한 사람이 다 하기 어려움. 분담·상호 지원 미정리.
- **역할 분담 제안**:

| 담당 영역 | 주요 업무 |
|-----------|-----------|
| AI·LLM 담당 | 모델 선정, 프롬프트 설계, 정보 추출, 질문 생성, 모델 성능 평가 |
| 데이터·RAG 담당 | 법령·공공자료 수집, 데이터 전처리, 벡터 DB 구축, 검색 성능 검증 |
| 서비스·백엔드 담당 | 웹앱 UI, 파일 업로드, API 연동, DB, 결과 리포트 및 사용자 흐름 구현 |

- OCR·VLM·규칙 엔진은 팀원 공동 지원. 모델 작업은 한 명 단독 금지, 최소 2명 검토.

## 3. MVP 개발 방향

### 3.1 MVP 핵심 흐름
1. 사용자가 현재 계약 단계 선택
2. 계약서·특약 PDF/이미지 업로드
3. OCR 또는 텍스트 추출
4. AI가 핵심 계약 정보 추출
5. 불명확·누락·확인 필요 항목 분류
6. 임대인·공인중개사에게 물어볼 질문 생성
7. 서명 전 체크리스트 제공
8. 계약 직후 권리 확보 행동 안내
9. 분석 근거·미완료 항목 포함 결과 리포트 생성

### 3.2 MVP 우선 구현 기능
- 모바일 웹앱 / PDF·이미지 업로드 / OCR·문서 텍스트 추출 / 핵심 계약 정보 구조화 / 계약 조항 쉬운 설명 / 확인 질문 생성 / 서명 전 체크리스트 / 법령·공공자료 기반 RAG / 결과 리포트 생성

### 3.3 제외·후순위 기능
- 네이티브 안드로이드 앱 / 로그인·복잡한 회원 관리 / 대규모 자체 모델 학습 / 모든 계약 문서 유형 지원 / 완전한 사기 여부 판정 / 실시간 등기부 자동 조회 / 공인중개사용 별도 화면 / 복잡한 대시보드·통계

## 4. 기술적 차별성 보완 방향

### 4.1 문서 간 교차 검증
- 계약서 임대인 vs 등기사항증명서 소유자 일치 여부
- 계약서 주소 vs 건축물대장 주소 일치 여부
- 계약 보증금 vs 확인·설명서 기재 일치 여부
- 계약서 vs 특약 상충 문구
- 계약 기간·입금 계좌 정보 불일치

### 4.2 위험 신호 규칙 엔진
- 소유자와 계약자가 다름 / 대리인 계약·위임장 확인 필요 / 보증금 반환 시점 불명확 / 수리·원상복구 책임 불명확 / 추가 근저당 설정 제한 조항 없음 / 선순위 보증금 확인 필요 / 계약금 입금 계좌 명의가 계약 상대와 다름

### 4.3 근거 기반 결과 제공
각 위험 신호에 함께 제공: 확인이 필요한 이유 / 관련 법령·공공 가이드 / 임대인·중개사에게 물어볼 질문 / 사용자가 해야 할 행동 / 확인되지 않은 경우 발생 가능한 문제 / 결과의 시급도.

### 4.4 모델 성능 비교
GPT/Gemini 단독 · GPT/Gemini + RAG · 로컬 단독 · 로컬 + RAG → 정보 추출 정확도·법률 근거 정확도·질문 생성 품질 비교.

## 8. 회의 요약

현재 프로젝트는 계약서 요약·질문 생성·체크리스트 제공 등 주요 기능이 범용 LLM만으로도 구현 가능하다는 한계가 있다. 해커톤 프로젝트로서 기술적 의미를 확보하려면 단순한 LLM 호출을 넘어 **계약·법률 데이터 기반 RAG, OCR·VLM 문서 분석, 문서 간 정보 교차 검증, 위험 신호 규칙 엔진, 모델 성능 비교**가 필요하다.

MVP는 스마트폰 계약 현장에서 사용할 수 있도록 **모바일 웹앱**으로 개발하며, **상용 LLM을 우선 활용하되 자체 데이터와 RAG로 정확성·근거성을 강화**한다. 팀은 다음 개발 단계 전 **데이터·모델·역할 분담·MVP 범위**를 우선 확정해야 한다.

## 9. 최종 모델·기술 선정표 (13단계, 팀 확정)

전량 상용 API 파이프라인 + 로컬 경량 처리. 개인정보는 로컬 비식별 후에만 API로 전송.

| # | 단계 | 모델/기술 | 위치 |
|---|------|-----------|------|
| 1 | 디지털 PDF 추출 | PyMuPDF / PDF.js | 로컬 |

## 13. 기술적 차별성 한 문장 (준비사항 #8)

> 범용 LLM은 못 하는 **계약서·등기사항증명서·건축물대장 3문서 교차검증**과 **등기 권리탐지 규칙 엔진**(근저당·압류·신탁)에 공식 법령 RAG 근거를 붙여, '안전/위험 단정' 대신 **근거 있는 확인 질문·행동**을 제공하는 첫 전월세 임차인용 계약 리스크 도우미.

핵심 레버 3개: 문서 교차검증 · 등기 권리탐지 규칙 엔진 · 개인정보 비식별 파이프라인.

## 14. 상용·로컬 운영 구조 (준비사항 #9)
