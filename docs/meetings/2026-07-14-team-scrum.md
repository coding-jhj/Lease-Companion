## 2. 한 줄 요약 · 핵심 결정

**한 줄 요약** — 계약서를 스마트폰으로 찍어 올리면, AI가 등기사항증명서·건축물대장과 교차검증하고 위험 신호를 잡아내 **근거 있는 확인 질문·행동**을 알려주는 모바일 웹앱. 개인정보는 로컬에서 비식별화한 뒤에만 외부 API로 전송한다.

**핵심 결정 5가지**
1. **서비스 형태** — 모바일 최적화 웹앱 (네이티브 앱 X, 회원가입·로그인 포함)
2. **LLM 전략** — 상용 API 우선(구조화 Gemini 3.5 Flash / 생성 GPT-5.6 Sol), 로컬 7B는 크리티컬 패스에서 제외
3. **문서 처리** — 텍스트 PDF는 직접 추출, 스캔·사진은 PaddleOCR-VL로 OCR+VLM 통합 처리
4. **차별화 3축** — ① 문서 교차검증 ② 등기 권리탐지 규칙 엔진 ③ 개인정보 비식별 파이프라인 (범용 LLM 단독으론 불가)
5. **개인정보 처리** — 로컬 토큰화 → API는 비식별 텍스트만 수신 → 결과 로컬 복원 → 세션 종료 시 전량 삭제

## 3. 준비사항 점검 (10개 중 9개 완료)

### ① 계약 데이터셋 및 공개 자료 조사 — ✅
- 공식 근거·서식 15종 확보(주택임대차보호법·시행령, 표준 임대차계약서, 중개대상물 확인·설명서, 모의 등기사항증명서·건축물대장 서식, HUG 안내 등).
- **합성 데이터 40쌍 제작**: dev 30 + test 10 (계약서·등기사항증명서·건축물대장 번들, 전부 비식별). 규칙별 발동/비발동 각 ≥10건, 개인정보 0, 누수 0 자동 검증 통과.
- 결론: 데이터는 "수집"이 아니라 "합성 제작" 문제 → 재료로 직접 제작 완료.

### ② LLM·VLM 후보 모델 목록과 선정 근거 — ✅
최종 모델·기술 선정표(13단계). 무거운 추론은 상용 API, 개인정보·최종 판정은 로컬.

| 단계 | 처리 | 모델/기술 | 위치 |
|------|------|-----------|------|
| 1~2 | PDF 추출 / OCR | PyMuPDF·PDF.js / PaddleOCR-VL-1.6 (VLM 통합) | 로컬 |
| 3~5 | 개인정보 탐지·토큰화 | privacy-filter-korean + 정규식 | 로컬 |
| 6 | 계약 필드 구조화 | Gemini 3.5 Flash | API |
| 7 | 추출값 확인·수정 | 사용자 UI | 로컬 |
| 8 | 위험 항목 최종 판정 | Python 규칙 엔진 | 로컬 |
| 9~10 | 근거 검색·재정렬 | gemini-embedding-001 + BM25 → Cohere rerank-v4.0-pro | API |
| 11 | 설명·질문·행동 생성 | GPT-5.6 Sol | API |
| 12~13 | 결과 복원 → 세션 종료 시 삭제 | 로컬 매핑표 | 로컬 |

로컬 7B는 크리티컬 패스 제외 → 상용 vs 로컬 성능비교 선택 실험.

### ③ OCR 및 PDF 처리 방식 — ✅
텍스트 레이어 확인 → 있으면 직접 추출(PyMuPDF·PDF.js), 없으면 OCR(PaddleOCR-VL) → 정규화 → 구조화. 표·레이아웃·도장은 PaddleOCR-VL의 VLM 통합으로 처리(별도 VLM 단계 없음).

### ④ RAG 데이터 구성안 — ✅
- 공식 근거 15종 코퍼스 + 규칙별 근거 매핑(rule_evidence_map).
- 파이프라인: 임베딩(gemini-embedding-001) + BM25 하이브리드 검색 → Cohere 리랭크 → 규칙 판정에 근거 부착.
- 검증: 규칙별 기대 근거 goldset 확보. RAG 적용 전후 성능 비교 예정. (출처 URL·시행일 일부 확인 필요)

### ⑤ 팀원별 역할 분담표 — ✅

| 담당 | 업무 | 담당자 |
|------|------|--------|
| AI·LLM | 모델 선정·프롬프트·정보 추출·질문 생성·성능 평가 | 정환주 (팀장) |
| 서비스·백엔드 | 웹앱 UI·업로드·API·DB·리포트·사용자 흐름 | 임명광 |
| 데이터·RAG | 법령·공공자료 수집·전처리·벡터 DB·검색 검증 | 김재현 |

공통: OCR·규칙 엔진 공동 지원, 모델 작업은 최소 2명 검토.

### ⑥ 최종 MVP 기능 목록 — ✅
회원가입·로그인 포함. 결과는 **종합 위험 점수(안전/위험/사기)를 쓰지 않고**, 판정별 상태 9종·시급도 5종으로 표현한다.

**사용자 흐름 8단계**
1. 회원가입·로그인
2. 계약 대시보드·계약 건 생성
3. 계약 상황 입력
4. 계약서·등기사항증명서·건축물대장 업로드
5. AI 추출값 확인·수정 (분석 전 사용자 검토)
6. 분석 — 상용 LLM 구조화 → 규칙 엔진 판정 → RAG 근거 → 상용 LLM 생성
7. 판정·원문 증거·공식 근거·질문·행동 리포트
8. 저장된 체크리스트·계약 직후 행동 관리

**판정 12개 (4영역)**
- **A. 당사자·목적물**: J01 임대인=등기 소유자 / J02 목적물 주소 일치 / J03 공동소유 확인 / J04 대리인·권한서류 / J05 계좌 명의=계약 상대
- **B. 금액·기간**: J06 보증금·월세·계약금·잔금 기재 / J07 금액 숫자·한글 표기 일치 / J08 지급일·입주일·기간 일관성 / J09 관리비 명확성
- **C. 책임·특약**: J10 보증금 반환 시점·조건 / J11 수리·원상복구 책임 / J12 본문-특약 상충
- **D. 행동 안내**: 단계별 미확인 항목·질문·서명 전 체크리스트·계약 직후 권리 확보 행동·보관 대상

### ⑦ 문서 교차 검증이 가능한 항목 목록 — ✅
규칙 R01~R10: ① 임대인=소유자 ② 주소 일치 ③ 근저당 ④ 압류·가압류 ⑤ 신탁 탐지 ⑥ 계좌 명의 ⑦ 등기 발급일 ⑧ 보증금 반환조건 ⑨ 수리 책임 ⑩ 권리변동 특약. (근저당·압류·신탁은 **존재 탐지**이지 위험 단정 아님)
- ⚠️ 12판정 중 **7개 미커버**: J03(공동소유)·J04(대리인 권한)·J06(금액 기재)·J07(금액 표기)·J08(날짜 일관성)·J09(관리비)·J12(본문-특약 상충) → 데이터·규칙 확장 필요.

### ⑧ 프로젝트의 기술적 차별성 한 문장 — ✅
> 범용 LLM은 못 하는 **계약서·등기사항증명서·건축물대장 3문서 교차검증**과 **등기 권리탐지 규칙 엔진**(근저당·압류·신탁)에 공식 법령 RAG 근거를 붙여, ‘안전/위험 단정’ 대신 **근거 있는 확인 질문·행동**을 제공하는 첫 전월세 임차인용 계약 리스크 도우미.

### ⑨ 상용 LLM과 로컬 모델의 운영 구조 — ✅
- 무거운 추론은 상용 API, **개인정보·최종 판정은 전부 로컬.**
- 개인정보 비식별 게이트: API로 보내기 전 로컬 토큰화 → 비식별 텍스트만 전송(6·9·11단계) → 로컬 복원 → 세션 종료 시 전량 삭제.
- Fallback: 상용 API 할당량·장애 시 Claude CLI·Codex CLI로 분기(비식별 게이트 동일 통과).
- 로컬 7B는 성능비교 실험(선택). 로컬 실행은 PaddleOCR-VL(0.9B)뿐 → GPU 부담 작음.

### ⑩ GPU 제공 여부와 학습 환경 — ❌ 미결
팀 확인 필요. 로컬 7B 크리티컬 패스 제외로 무거운 GPU는 불필요, 성능비교 실험 진행 여부와 연결.
