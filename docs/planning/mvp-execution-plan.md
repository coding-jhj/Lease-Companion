# 실전 계약 점검 MVP — 실행 계획 (진행 순서·역할 분담)

> 기준일: 2026-07-16 팀 합의.
> 이 문서는 일정 약속이 아니라 **단계·선행 조건·완료 기준**으로 진행을 관리한다. 주차·일수 표기는 사용하지 않는다.
> 범위: **1단계 = R01~R10 기반 실전 계약 점검 MVP 완성 → 2단계 = J01~J12 전체 판정 확장.** J01~J12는 영구 제외가 아니라 후속 확장이다. R과 J는 같은 번호 체계로 합치지 않는다.

## 1. 확정된 결정 (2026-07-16)

기술 결정 상세와 이유는 ADR을 단일 기준으로 한다.

- 플랫폼: [`../decisions/2026-07-16-mvp-platform-stack.md`](../decisions/2026-07-16-mvp-platform-stack.md) — PostgreSQL / JWT Bearer + bcrypt 계열 / React + Vite + TypeScript / Chroma 로컬 모드 / 현재 로컬 실행(운영 배포 플랫폼 미정)
- 통합 스키마: [`../decisions/2026-07-16-shared-pydantic-schema.md`](../decisions/2026-07-16-shared-pydantic-schema.md) — `ai/src/lease_companion_ai/schemas/`의 Pydantic 모델 단일 원본, Backend import 재사용, `user_corrected_value` / `verification_status` / 3등급 confidence(`추출됨`·`불확실`·`실패`) / nullable `page`·`text`

## 2. 역할 분담

| 담당 | 영역 | 책임 |
|---|---|---|
| 담당 A | AI·데이터 | 통합 스키마 코드화, 추출 메타데이터, R01~R10 회귀 검증, 공식자료 RAG(청킹·검색·rerank), 근거 제한 생성, Guardrail. (이 저장소의 주 작업자가 담당) |
| 담당 B | Backend·저장 | PostgreSQL 셋업, 회원·계약 건·계약 상황 API, 문서 저장·이력, 모의 등기 연결(`registry-link`), 분석 결과·체크리스트·행동·리포트 저장·재조회, 접근 통제 |
| 담당 C | Frontend | React + Vite + TypeScript 셋업, 8단계 화면, mock 우선 개발, 확인 우선순위 UI·접근성, 실제 API 연결, E2E |

## 3. 단계별 진행 (선행 조건 → 완료 기준)

### 0단계 — 공통 기반

| 담당 | 할 일 | 완료 기준 |
|---|---|---|
| B | PostgreSQL 설치·연결 셋업(Docker 또는 로컬) + SQLAlchemy 연결 확인 | 팀 공용 DB에 연결·조회가 된다 |
| C | React + Vite + TypeScript 프로젝트 셋업(`frontend/`) + 라우팅 뼈대 | 라우팅이 동작하는 빈 앱이 뜬다 |

선행 조건: 없음 (ADR 확정으로 즉시 시작 가능).

### 1단계 — 기초 공사 (병렬)

| 담당 | 할 일 | 선행 조건 | 완료 기준 |
|---|---|---|---|
| A | **통합 스키마 코드화** — 합의된 표(ExtractedField·판정 코어)를 `ai/src/lease_companion_ai/schemas/`에 Pydantic 모델로 작성, 기존 코드에 어댑터 연결 | ADR 2개 | 교체 전·후 R01~R10 회귀 테스트 결과 동일 |
| B | **회원 API**(signup/login/me — JWT 발급) + **계약 건 API**(생성·목록·상세) + **계약 상황 입력 API** | 0단계 B | 로그인해서 계약 건을 만들고 다시 조회할 수 있다 |
| C | **8단계 화면 뼈대** — mock 데이터로 개발 (API 목록의 응답 형태 기준) | 0단계 C | 화면 흐름 전체가 클릭으로 이어진다 |

경계 규칙: **A의 스키마 확정 전에는 B가 추출 데이터 저장 모델을, C가 추출값 확인·수정 화면을 만들지 않는다.** 그동안 B는 회원·계약 건, C는 로그인·대시보드 등 스키마 무관 영역을 진행한다.

### 2단계 — 본 공사 (병렬)

| 담당 | 할 일 | 선행 조건 | 완료 기준 |
|---|---|---|---|
| A | **RAG 구축** — 공식 자료 검증·청킹 후 Chroma(로컬) 적재, gemini-embedding-001 + BM25 검색, Cohere rerank, 규칙 결과별 근거 연결 | 1단계 A | 공식 근거가 연결되며, 근거가 없으면 판정은 그대로 두고 `evidence_sources=[]` 반환 |
| B | **문서 업로드 저장**(계약 건 연결·이력) + **모의 등기 연결**(`registry-link`, `case_id` 기준) + **분석 결과 저장·재조회**(버전·재분석 이력) | 1단계 A·B | 분석 결과가 저장되고 재로그인 후 다시 열린다 |
| C | **확인 우선순위 UI**(반드시 확인·확인 권장·일반 확인 — 색+문구+아이콘, 접근성) + 로딩·오류·빈 상태 + 대시보드 리포트 재진입 | 1단계 C | 결과 화면이 3그룹 UI로 표시된다 |

### 3단계 — 연결 공사 (병렬)

| 담당 | 할 일 | 선행 조건 | 완료 기준 |
|---|---|---|---|
| A | **질문·행동 생성 + Guardrail** — GPT-5.6 Sol로 확인 질문·체크리스트·행동 문구 생성. 단정 표현·근거 없는 출력 차단. 규칙 판정 불변 | 2단계 A (근거 없이 생성부터 만들면 원칙 위반) | 생성 출력이 Guardrail을 통과한 것만 저장된다 |
| B | **체크리스트·계약 직후 행동 API**(상태 저장·재조회) + 접근 통제(타인 계약 건 차단) | 2단계 B | 다른 계정으로는 남의 계약 건이 보이지 않는다 |
| C | **mock → 실제 API 교체** | 2단계 B·C | 실제 서버로 전체 흐름이 동작한다 |

### 4단계 — 통합 검증 (공동)

선행 조건: 1~3단계 완료. **CASE-001**로 처음부터 끝까지 실행한다.

- [ ] 회원가입 → 로그인 → 계약 건 생성 → 상황 입력 → 문서 업로드 → 추출 → 수정·확인 → 분석 → 리포트 → 체크리스트
- [ ] 로그아웃 후 재로그인해도 리포트가 다시 열리는가
- [ ] 다른 계정으로 로그인하면 남의 계약 건이 안 보이는가
- [ ] Gemini 키 없이(폴백 경로) 앱이 죽지 않는가
- [ ] 로그에 이름·주소 같은 개인정보가 찍히지 않는가
- [ ] "안전합니다" 같은 단정 문구가 화면 어디에도 없는가

### 5단계 — J01~J12 전체 판정 확장 (1단계 MVP 완성 후)

선행 조건: 4단계 통과.

- R01~R10의 `result_type(judgment|fact_flag)`·`triggers_actions`는 canonical v1.1.0에 구현 완료했다. J 확장에서는 이 계약을 유지하며 미구현 판정(J03·J04·J06·J07·J08·J09·J12)을 추가한다.
- 통합 스키마는 **필드 추가**로 확장하고 기존 R01~R10 필드의 이름·의미를 바꾸지 않는다(하위 호환 — ADR 참조).
- R↔J 매핑은 [`../data/judgment-spec.md`](../data/judgment-spec.md)에서 관리한다.
- 완료 기준: J01~J12가 판정별 허용 상태·시급도·입력 부족 정책에 맞게 실행되고 goldset 회귀를 통과한다.

## 4. 공동 규칙

1. A의 스키마 확정 전에 추출 데이터를 저장(B)하거나 그리는(C) 코드를 만들지 않는다.
2. API 요청·응답 형태 변경은 반드시 3인 공유 후 진행한다.
3. 새 API 추가는 자유, 기존 API 변경은 신중하게.
