# 최소 1차 MVP 범위

> 이 문서는 전체 MVP([`mvp-scope.md`](mvp-scope.md))를 바꾸는 문서가 아니라, 전체 MVP로 가기 위한 **첫 번째 구현 슬라이스**를 정의한다. 전체 판정의 단일 기준은 [`../data/judgment-spec.md`](../data/judgment-spec.md)의 J01–J12다.

## 목표

계약 문서를 업로드한 사용자가 구조화된 추출값을 직접 확인·수정한 뒤, 소수의 결정론적 규칙과 공식 근거가 연결된 행동 중심 리포트를 받는 흐름을 먼저 검증한다.

이 단계에서도 `안전`·`위험`·`전세사기 가능성` 같은 종합 판정은 제공하지 않는다.

## 사용자 흐름 (6단계)

1. **문서 입력**: 계약서 PDF 업로드
2. **정보 추출**: 계약서 추출 스키마에 따라 LLM이 정해진 JSON 형태로 반환
3. **사용자 검증**: 추출값 수정 화면, 미확인 값 표시, 확인 완료
4. **규칙 엔진**: 우선순위가 높은 핵심 규칙 약 10개 실행
5. **공식 근거 연결**: 규칙별 공식 근거 1~3개, 근거 요약, 원문 URL 연결
6. **리포트**: 시급도, 확인 이유, 질문, 권장 행동, 공식 근거, 완료 여부 제공

전체 MVP의 회원·계약 건 관리와 저장·재조회는 후속 통합 대상이다. 다만 1차 구현도 나중에 `ContractProject`에 귀속시킬 수 있도록 문서·추출·분석 식별자 경계를 유지한다.

## 단계별 완료 기준

### 1. 문서 입력

- 1차 필수 입력은 계약서 PDF 1개다.
- 파일 형식·크기·페이지 수 제한값은 미정(TODO)이며, 임의로 확정하지 않는다.
- 스캔 PDF·사진 OCR은 상용 LLM Gemini 3.5 Flash VLM 통합으로 확정(2026-07-14 변경, → [`../decisions/2026-07-14-ocr-gemini-integration.md`](../decisions/2026-07-14-ocr-gemini-integration.md)). 최소 데모에 구현 완료: 디지털 PDF는 PyMuPDF 텍스트 레이어, 텍스트 레이어가 없는 스캔 PDF·이미지(jpg·png)만 OCR 폴백. OCR 충실도·비용·지연은 미측정(TODO).
- 등기사항증명서는 아래 교차검증 규칙에 필요하다. 계약서만 받는 버전에서는 관련 규칙을 추측 실행하지 않고 `확인 불가`로 반환한다.

### 2. 정보 추출

- LLM 출력은 자유 텍스트가 아니라 버전이 명시된 JSON 스키마를 따른다.
- 각 필드는 값뿐 아니라 원문 위치, 확인 상태, 추출 실패 사유를 구분할 수 있어야 한다.
- 추출기는 판정하지 않는다. 규칙 결과·설명·질문·행동을 추출 JSON에 섞지 않는다.
- 값이 없거나 읽히지 않으면 지어내지 않고 `null`과 사유를 반환한다.
- 스키마 초안: [`../../data/schemas/minimum-mvp-extraction-v1.schema.json`](../../data/schemas/minimum-mvp-extraction-v1.schema.json)

### 3. 사용자 검증

- 사용자는 각 추출값을 수정할 수 있다.
- `미확인` 값은 화면에서 구분되어야 한다.
- `확인 완료` 전에는 규칙 분석을 시작하지 않는다.
- 원 추출값과 사용자 수정값(`user_corrected_value`)을 별도로 보존한다. 수정값이 규칙 엔진의 입력이다.
- 화면은 React + Vite + TypeScript(2026-07-16 확정)로 구현한다. 데이터 형태는 canonical Pydantic 스키마 계약을 따른다(→ [`../decisions/2026-07-16-shared-pydantic-schema.md`](../decisions/2026-07-16-shared-pydantic-schema.md)).

### 4. 핵심 규칙

핵심 규칙은 J01–J12를 대체하거나 새 종합 판정을 추가하지 않는다. 등기 권리사항과 발급일은 J01–J12의 판정이 아니라 **사실 플래그와 행동 트리거**로 기록한다.

> **명명 정리**: 아래 10개 규칙은 구현·리포트·`data/rules/rule_spec.csv`에서 **R01~R10** 식별자로 부른다. R은 최소 MVP의 1차 규칙 번호이고 J는 전체 MVP 판정 체계다 — 같은 축이 아니며, R 규칙은 아래 표처럼 J 판정에 연결되거나 D영역 사실 플래그로 기록된다.

| 1차 규칙 | 분류 | 기존 판정 연결 | 필요한 입력 | 입력 부족 시 |
|----------|------|----------------|-------------|--------------|
| 임대인 이름 불일치 | 교차검증 | J01 | 계약서 + 등기사항증명서 | `확인 불가` |
| 목적물 주소 불일치 | 교차검증 | J02 | 계약서 + 등기사항증명서 | `확인 불가` |
| 근저당권 기재 존재 | 사실 플래그·행동 트리거 | D영역 | 등기사항증명서 | 미확인 항목 생성 |
| 압류·가압류 기재 존재 | 사실 플래그·행동 트리거 | D영역 | 등기사항증명서 | 미확인 항목 생성 |
| 신탁등기 기재 존재 | 사실 플래그·행동 트리거 | D영역 | 등기사항증명서 | 미확인 항목 생성 |
| 입금 계좌 명의 불일치 | 문서 내부 비교 | J05 | 계약서 임대인·계좌 명의 | 값 없으면 `확인 필요` |
| 등기사항증명서 발급일 확인 | 문서 최신성 플래그 | D영역 | 등기사항증명서 발급일 + 분석 기준일 | 기준 기간 미정(TODO) |
| 보증금 반환 시점·조건 누락 | 명확성 | J10 | 계약서 본문·특약 | 조항 없으면 `미기재` |
| 수리·원상복구 책임 누락 | 명확성 | J11 | 계약서 본문·특약 | 조항 없으면 `미기재` |
| 권리변동 관련 특약 누락 | 사실 플래그·행동 트리거 | D영역 | 계약서 특약 | 확인 질문 생성 |

주의:

- 근저당권·압류·가압류·신탁등기의 **존재만으로 계약의 안전·위험·적법 여부를 판정하지 않는다.** 확인 이유·질문·증빙 행동으로 연결한다.
- 등기사항증명서 최신성 판단에 사용할 일수 기준은 공식 근거와 기획 결정 전까지 TODO로 둔다.
- 권리변동 관련 특약의 누락은 본문–특약 상충(J12)과 다르므로 J12로 잘못 기록하지 않는다.

### 5. 공식 근거

각 규칙 또는 행동 트리거에는 공식 근거 1~3개를 연결한다.

필수 메타데이터:

- 자료명
- 발행기관
- 조문·항목
- 시행일 또는 게시일
- 원문 URL
- 수집일
- 사용자용 근거 요약

공식 근거는 설명을 뒷받침하지만 규칙 판정을 변경하지 않는다. 실제 자료와 URL은 공식 출처 확인 후 [`../../data/rag/metadata/`](../../data/rag/metadata/)에 등록한다(TODO).

### 6. 리포트

규칙별 리포트 카드는 최소한 다음 필드를 가진다.

| 필드 | 의미 |
|------|------|
| `status` | J01–J12에 해당하면 허용된 공통 상태, 사실 플래그이면 별도 사실 값 |
| `urgency` | 공통 시급도 5개 중 하나 |
| `reason` | 무엇을 확인해야 하는지에 대한 쉬운 설명 |
| `question` | 임대인·공인중개사 등에게 물을 질문 |
| `recommended_actions` | 사용자가 확인·요청·보관할 행동 |
| `evidence_sources` | 공식 근거 요약과 원문 URL |
| `completed` | 사용자의 확인·행동 완료 여부 |

`completed`는 사용자의 진행 상태일 뿐 판정 상태를 바꾸지 않는다.

## 입력 의존성 및 1차 출시 경계

계약서 PDF만 업로드하는 흐름과 10개 규칙 전체 실행은 동시에 성립하지 않는다.

- **계약서만 있을 때 실행 가능**: 계좌 명의 비교, 반환 조건, 수리 책임, 권리변동 특약
- **등기사항증명서가 있어야 실행 가능**: 이름·주소 교차검증, 근저당권, 압류·가압류, 신탁등기, 발급일

따라서 1차 개발 시 다음 중 하나를 기획 결정해야 한다.

1. 계약서 PDF만 받고 등기 의존 규칙은 `확인 불가`로 노출한다.
2. 등기사항증명서 PDF를 선택 입력으로 추가해 10개 규칙 전체를 실행한다.

**결정(2026-07-14 데모): 옵션 2 채택** — 계약서·등기사항증명서 **둘 다 필수 입력**으로 구현(`backend/app/schemas/minimum_mvp.py`). 규칙 엔진은 값이 없는 항목을 `확인 불가`로 반환한다.

## 제외 범위

- 로컬 7B 파인튜닝·모델 다운로드
- 회원·대시보드·체크리스트 장기 저장의 실제 구현 (전체 MVP·실행 계획에서 진행)
- 종합 안전·위험 점수 및 법률적 결론
- 등기 권리금액을 이용한 재무적 위험도 계산

> 기술 스택은 이 문서 최초 작성 시점과 달리 대부분 확정되었다: LLM·OCR·임베딩(선정표, 2026-07-14), DB PostgreSQL·인증 JWT Bearer+bcrypt 계열·프론트엔드 React+Vite+TypeScript·벡터 DB Chroma 로컬 모드(2026-07-16, → [`../decisions/2026-07-16-mvp-platform-stack.md`](../decisions/2026-07-16-mvp-platform-stack.md)).
