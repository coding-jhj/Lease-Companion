# AGENTS.md — 슬기로운 계약생활 (lease-companion)

이 파일은 Codex·Claude Code 등 모든 AI 에이전트의 **단일 기준 지시서(Source of Truth)**다. 작업 전 반드시 읽는다.

## 프로젝트 목적

회원 기반 모바일 웹앱에서 사용자가 **계약 건**과 **계약 문서**를 관리하고, **상용 LLM(구조화)**이 비식별 텍스트에서 임대차 조항 유형과 불명확성 후보를 구조화한다. **Python 규칙 엔진**은 계약서와 관련 문서의 값을 교차검증하고, **공식자료 RAG**와 **상용 LLM(생성)**은 판정 근거·쉬운 설명·확인 질문·사용자 행동을 생성한다. (로컬 7B는 MVP 크리티컬 패스에서 **제외** — 선택적 성능비교 실험. 2026-07-14 최종 모델·기술 선정표.)

- **서비스명**: 슬기로운 계약생활
- **대상 사용자**: 첫 전월세 계약을 준비하는 2030 청년 임차인
- **대상 계약**: 주거용 전세 / 보증부 월세 / 일반 월세
- **제외 범위**: 부동산 매매, 상가·사무실 임대차, 경매·공매, 소송, 계약 종료·보증금 반환 분쟁, 거주 중 하자 대응

분석은 단일 세션으로 끝나지 않는다. 로그인한 사용자의 **계약 건 단위**로 분석 결과·체크리스트·계약 직후 행동 상태를 저장하고 재조회한다.

## 사용자 흐름 (기준 8단계)

1. 회원가입·로그인
2. 계약 대시보드·계약 건 생성
3. 계약 상황 입력
4. 계약서·등기 등 문서 업로드
5. 추출값 확인·수정
6. 상용 LLM 구조화·규칙 엔진·RAG·상용 LLM 생성 분석 (로컬 7B는 선택적 비교 실험)
7. 판정·원문 증거·공식 근거·질문·행동 리포트
8. 저장된 체크리스트·계약 직후 행동 관리

상세: [`docs/planning/user-flow.md`](docs/planning/user-flow.md). AI 추출값은 분석(6단계) 전에 사용자가 **확인·수정**할 수 있어야 한다.

## 아키텍처 (기준: 2026-07-15 아키텍처 다이어그램)

구현은 아래 5개 블록과 흐름을 기준으로 진행한다.

| 블록 | 구성 요소 |
|------|-----------|
| 사용자 화면 | 청년 임차인 웹·모바일 사용자 — 계약 상황 입력 / 계약서 PDF 업로드 / 추출값 확인·수정 / 위험 신호·확인 리포트 |
| 백엔드 서버 | 문서 처리 API / 계약 건 관리 API / 분석·리포트 API |
| AI 분석 파이프라인 | PDF 텍스트 추출(OCR·PDF Parser) → LLM 정보 추출(JSON Schema) → 규칙 엔진(핵심 규칙 10개) → RAG 기반 설명 생성 → 질문·체크리스트·행동 안내 생성 |
| 데이터·지식 저장소 | 문서 저장소 / 계약 정보 DB / 분석 결과·이력 / 법령·공식 자료 Vector DB / 모의 등기부 데이터 |
| 사용자 제공 결과 | 신호등 요약(반드시 확인·확인 권장·일반 확인) / 임대인·중개사에게 물어볼 질문 / 서명 전 체크리스트 / 계약 직후 행동 안내 / 임차권 방어 리포트 |

### 데이터 흐름

1. 사용자가 계약 상황을 입력하고 계약서 PDF를 업로드한다 → 문서 처리 API → 문서 저장소.
2. AI 파이프라인: PDF 텍스트 추출 → LLM 정보 추출(고정 JSON Schema) → **사용자 추출값 확인·수정** → 규칙 엔진.
3. 규칙 엔진은 모의 등기부 데이터를 교차검증에 사용하고, RAG(법령·공식 자료 Vector DB)가 근거·설명을, 생성 단계가 질문·체크리스트·행동 안내를 만든다.
4. 계약 건은 계약 정보 DB에, 분석 산출물은 분석 결과·이력에 저장하고, 분석·리포트 API가 사용자 화면의 위험 신호·확인 리포트로 제공한다.

### 신호등 표현 (사용자 화면 전용)

`반드시 확인` · `확인 권장` · `일반 확인` 3단계는 **화면 표시용 확인 우선순위**다.

- 내부 판정 상태 9개·시급도 5개를 대체하지 않는다. 시급도→신호등 매핑(2026-07-16 확정, 화면 표시 전용 — 종합 안전·위험 판정 아님): `즉시 확인`·`분석 불가` → **반드시 확인** / `계약 전 확인`·`계약 직후 조치` → **확인 권장** / `참고` → **일반 확인**. 화면 구현 규칙은 [`frontend/AGENTS.md`](frontend/AGENTS.md), 판정 상세는 [`docs/data/judgment-spec.md`](docs/data/judgment-spec.md).
- "위험 신호·확인 리포트"·"임차권 방어 리포트"의 **위험 신호는 항목별 확인 필요 신호**를 뜻한다. 종합 안전·위험·사기 점수 금지 원칙은 그대로 유지된다.

### 판정 규모 표기

- 규칙 엔진 **핵심 규칙 10개** = 현행 최소 MVP 규칙 엔진(R01–R10) 기준.
- MVP 목표 판정은 4영역 12항목(J01–J12, 아래 판정 범위)이며, 규칙 10개→12판정 확장 매핑은 판정 명세에서 관리한다.

### 구현 순서 (2026-07-16 확정)

1. **1단계**: R01~R10 기반 실전 계약 점검 MVP를 먼저 완성한다.
2. **2단계**: J01~J12 전체 판정으로 확장한다. J01~J12는 영구 제외가 아니라 후속 확장이다.

R과 J는 서로 다른 축이며 같은 번호 체계로 합치지 않는다. 일정은 주차·일수가 아니라 단계·의존성·완료 조건으로 관리한다.

### 통합 스키마와 식별자 (2026-07-16 확정)

- **Canonical runtime schema**: `ai/src/lease_companion_ai/schemas/`의 **Pydantic 모델 하나**가 런타임 통합 스키마의 단일 원본이다. Backend는 이 공통 타입을 import해 사용하고 같은 도메인 타입을 중복 정의하지 않는다. JSON Schema는 손으로 쓰지 않고 Pydantic에서 생성한다. → [`docs/decisions/2026-07-16-shared-pydantic-schema.md`](docs/decisions/2026-07-16-shared-pydantic-schema.md)
- **식별자 구분**: `contract_id`(실제 사용자 계약 건) / `case_id`(CASE-001 같은 합성·평가 사례) / `document_id`(업로드 문서) / `input_snapshot_id`(사용자 확인 완료 입력) / `analysis_run_id`(분석 실행 1회). **`contract_id`와 `case_id`를 같은 의미로 사용하지 않는다.**

## PoC / MVP 관계

PoC와 MVP는 서로 다른 기능 목록이 아니다. **PoC는 기술 검증 단계**, **MVP는 검증된 기술을 통합한 실제 사용자 서비스**다.

- **PoC (기술 검증)**: 계약서·등기사항증명서 샘플 입력 → PDF·OCR 입력 처리 → 필드 추출·정규화 → 상용 LLM 구조화(로컬 7B는 선택적 비교) → 핵심 판정 6개 → 규칙 기반 문서 교차검증 → RAG 공식 근거 → 구조화 JSON 출력 → 모델 비교평가
- **MVP (사용자 서비스)**: 회원가입·로그인 → 계약 건 생성 → 문서 업로드 → 추출값 확인·수정 → 12개 판정 → 등기사항증명서 교차검증 → 공식 근거·질문·행동 리포트 → 결과 저장·재조회 → 체크리스트·계약 직후 행동 상태 관리

상세: [`docs/planning/poc-scope.md`](docs/planning/poc-scope.md), [`docs/planning/mvp-scope.md`](docs/planning/mvp-scope.md). 문서·코드에서 두 범위를 혼용하지 않는다.

## 판정 범위 (4영역 12항목)

A. 계약 당사자·목적물: J01 임대인=등기 소유자 / J02 목적물 주소 일치 / J03 공동소유 확인 / J04 대리인·권한서류 확인 / J05 입금 계좌 명의=계약 상대
B. 금액·기간: J06 보증금·월세·계약금·잔금 기재 상태 / J07 금액 숫자·한글 표기 일치 / J08 지급일·입주일·계약기간 일관성 / J09 관리비 금액·포함항목 명확성
C. 책임·특약: J10 보증금 반환 시점·조건 명확성 / J11 수리·원상복구 책임 명확성 / J12 본문-특약 상충 가능성
D. 행동 안내: 단계별 미확인 항목, 계약금 입금 전 질문, 서명 전 체크리스트, 계약 직후 권리 확보 행동, 보관 대상(계약서·이체내역·대화 기록)

상세: [`docs/data/judgment-spec.md`](docs/data/judgment-spec.md).

## 컴포넌트 책임 구분

- **상용 LLM (구조화)**: 비식별 텍스트에서 계약 필드·조항 유형·명확성 후보를 고정 JSON으로 구조화. **최종 판정을 내리지 않는다.** (선정표: Gemini 3.5 Flash)
- **Python 규칙 엔진**: 문서 내부 판정과 문서 교차검증의 **명시적 최종 판정**. LLM·로컬 모델이 규칙 결과를 임의로 변경하지 않는다.
- **RAG**: 공식 자료 기반 **근거 검색**. 판정을 내리지 않는다. 근거가 없으면 `확인 불가` 또는 `확인 필요`로 반환한다.
- **상용 LLM (생성)**: 저신뢰 결과 재검토, 근거 기반 쉬운 설명·질문·행동 생성. 규칙 판정을 바꾸지 않는다. (선정표: GPT-5.6 Sol)
- **로컬 7B 모델 (선택)**: **MVP 크리티컬 패스 제외**(선정표). 상용 vs 로컬 성능 비교 병렬 실험용. 조항 유형·명확성 후보 분류를 하되 최종 판정·규칙 결과를 변경하지 않는다.
- **guardrails**: 단정 표현·근거 없는 출력 차단.
- **routing**: 단계별 처리 모델·fallback 선택.
- **backend**: 전체 작업 오케스트레이션과 저장.

### 로컬 7B가 하지 않는 것
최종 법률 결론 / 규칙 엔진의 명시적 비교 결과 변경 / 공식 근거 없는 사용자 행동 확정.

### AI가 하지 않는 것
전세사기 여부 확정 / 계약 안전 여부 확정 / 적법·위법 확정 / 계약 체결 추천·거절 / 도장·서명·신분증 진위 판정 / 법률 상담 대체.

## 원칙 (모든 에이전트 준수)

- AI는 계약 가능·안전·전세사기·법률적 결론을 **단정하지 않는다.** 확인 항목·질문·체크리스트·증빙 행동만 제공한다.
- 추출값과 생성값을 구분한다. 규칙 엔진 결과를 LLM이 임의로 변경하지 않는다. RAG는 판정하지 않는다. 근거 부족 시 정보 부족·확인 필요로 반환한다.
- 실제 개인정보·계약 문서·모델 가중치·체크포인트를 Git에 커밋하지 않는다. 비식별·합성 데이터만 커밋한다.
- 기존 파일과 사용자 변경사항을 임의로 삭제·덮어쓰지 않는다.
- 요청 범위를 벗어난 기능·기술을 임의로 추가하지 않는다.
- **미정 기술은 임의 확정하지 않고 README/문서에 `TODO`로 표시한다.** 아래 목록 참조.
- 프롬프트는 버전 관리한다. Guardrail을 통과한 출력만 저장한다.
- 단계별 평가 데이터(train/validation/test)를 분리한다. 목표 성능 수치는 실제 측정 전 임의로 만들지 않는다.
- 결정 기록(`docs/decisions/`)과 회의록(`docs/meetings/`)을 유지한다.
- 작업 전 적용 범위의 `AGENTS.md`와 관련 문서를 확인한다. 변경 후 관련 테스트·문서를 확인한다.
- 테스트하지 않은 내용을 테스트 완료라고 쓰지 않는다.
- `git commit` / `git push`는 사용자의 명시적 요청이 있을 때만 수행한다.
